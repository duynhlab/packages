# File & config reference

Every file the `duynhlab` RPM puts on (or generates on) a host, in one place:
path, purpose, owner/mode, and **what creates it** — the RPM payload itself,
a `%post` scriptlet, or you. Ground truth is `packages/rpm/duynhlab.spec`
(`%install`/`%files`), `packages/rpm/lib/init-service.sh` and
`packages/rpm/lib/password-generator.sh`; verify any row on a live host with
`rpm -qlv duynhlab`.

**Created-by legend**: `payload` = shipped inside the RPM (`%files`, replaced
on upgrade) · `init-service` = dropped by `%post` via
`/opt/duynhlab/lib/init-service.sh`, **copy-if-missing, never overwrites** ·
`password-generator` = generated by `%post` via
`/opt/duynhlab/lib/password-generator.sh`, version-gated · `operator` = you
create it; the RPM never touches it.

| user | `/opt/duynhlab/user/bin/user-service` | 8002 | — | `duynhlab_user` | `duynhlab-user.service` |
| product | `/opt/duynhlab/product/bin/product-service` | 8003 | — | `duynhlab_product` | `duynhlab-product.service` |
| cart | `/opt/duynhlab/cart/bin/cart-service` | 8004 | — | `duynhlab_cart` | `duynhlab-cart.service` |

Per-service audit files next to each binary (`/opt/duynhlab/<svc>/`):
`BINARY_VERSION` (built version), `SCHEMA_VERSION` (highest embedded migration
— informational only, nothing blocks startup), `README.md`. The exact source
commits of all 9 components are in `/opt/duynhlab/etc/manifest`.

## 3. Rest of the payload under `/opt/duynhlab`

Everything here is replaced wholesale on upgrade — never edit it in place.

| Path | What | Created by |
|---|---|---|
| `lib/init-service.sh` | `%post` scriptlet: creates dirs, drops configs below | payload |
| `lib/password-generator.sh` | `%post` scriptlet: renders `secret-tpl/` → `/etc/duynhlab/<svc>.env` | payload |
| `secret-tpl/<svc>.env.tpl` (×8) | Env templates with `__DB_PASSWORD__` placeholder | payload |
| `nginx/duynhlab.conf` | Reverse-proxy vhost template (API + frontend) | payload |
| `nginx/duynhlab-frontend.conf` | Optional standalone frontend vhost (NOT auto-installed) | payload |
| `valkey/duynhlab.conf` | Valkey snippet template | payload |
| `postgresql/duynhlab-tuning.conf` | PostgreSQL tuning template | payload |
| `postgresql/bootstrap.sql` | DB/role bootstrap template, applied by `duynhdb bootstrap` | payload |
| `logrotate/duynhlab-services`, `logrotate/duynhlab-nginx` | logrotate rule templates | payload |
| `etc/services.yaml` | Service registry master copy | payload |
| `etc/env-global.properties` | Global defaults master copy (`DUYNHLAB_VERSION`, `LOG_LEVEL`, `DB_HOST`…) | payload |
| `etc/manifest` | Composition audit: 9 component commits/versions in this build | payload |

## 4. Config in `/etc/duynhlab` (survives upgrades)

| Path | Owner / mode | Created by | Notes |
|---|---|---|---|
| `services.yaml` | `root:root 0644` | init-service | Copy of the registry; read by `duynhctl`/`duynhdb`. Copy-if-missing — your edits stick |
| `env-global.properties` | `root:root 0644` | init-service | Shared defaults, loaded first by every unit |
| `<svc>.env` (×8) | `root:duynhlab 0640` | password-generator | Random 32-char `DB_PASSWORD`, `PORT`, `GRPC_PORT`…; `%ghost` — generated once, **never overwritten** |
| `<svc>.override` | yours | operator | Optional; loaded last, wins. The RPM never creates or touches it |
| `secret_version.properties` | `root:duynhlab 0640` | password-generator | `secretVersion=N` — highest completed secret block; lets future releases add new secrets without touching old ones |

How the three env layers combine → [`003-operations.md` §5](003-operations.md#5-configuration).

## 5. Config dropped into system directories

Installed by `init-service.sh` on `%post`, **only if the destination file does
not already exist** (your versions are never overwritten). Valkey/PostgreSQL
drops are best-effort: skipped silently when the target directory is absent.

| Destination | Source template | Mode / owner | Condition |
|---|---|---|---|
| `/etc/nginx/conf.d/duynhlab.conf` | `/opt/duynhlab/nginx/duynhlab.conf` | `0644 root:root` | `/etc/nginx/conf.d` exists |
| `/etc/valkey/conf.d/duynhlab.conf` (fallback `/etc/valkey/duynhlab.conf`) | `/opt/duynhlab/valkey/duynhlab.conf` | `0640 root:root` | first existing dir wins |
| `<pg-confdir>/duynhlab-tuning.conf` | `/opt/duynhlab/postgresql/duynhlab-tuning.conf` | `0644 postgres:postgres` | first of `/etc/postgresql/conf.d`, `/var/lib/pgsql/data/conf.d`, `/var/lib/pgsql/{16,15,14}/data/conf.d` |
| `/etc/logrotate.d/duynhlab-services`, `/etc/logrotate.d/duynhlab-nginx` | `/opt/duynhlab/logrotate/*` | `0644 root:root` | `/etc/logrotate.d` exists |

## 6. systemd units (`/usr/lib/systemd/system/`)

| Unit | Purpose |
|---|---|
| `duynhlab-platform.target` | Operator entry point — starts/stops all 8 backends |
| `duynhlab-infra.target` | Ordering against external `nginx`/`postgresql`/`valkey` (does not own them) |
| `duynhlab-<svc>.service` (×8) | One per backend (see table in §2); `PartOf=` the platform target |

## 7. Runtime state

| Path | Owner / mode | Created by | Notes |
|---|---|---|---|
| `/var/log/duynhlab/` | `duynhlab:duynhlab 0755` | init-service | Log root |
| `/var/log/duynhlab/<svc>/` (×8) | `duynhlab:duynhlab 0755` | init-service | Per-service app logs (journald is primary) |
| `/var/log/duynhlab/nginx/` | `root:root 0755` | init-service | nginx access/error logs (vhost config points here) |
| `/var/log/duynhlab/version.log` | root | `%post` | Append-only install history: `<version> installed on <ts> (op=…)`. Survives uninstall |
| `/var/lib/duynhlab/` | `duynhlab:duynhlab 0750` | init-service | State/cache root |

## 8. What survives an upgrade vs. what is replaced

- **Replaced every upgrade**: all of `/opt/duynhlab/` (binaries, templates,
  scripts, manifest), systemd units, `/usr/bin` symlinks, bash completion.
- **Preserved**: everything in `/etc/duynhlab/` (§4), every config dropped
  copy-if-missing into system dirs (§5), `/var/log/duynhlab/` and
  `/var/lib/duynhlab/` contents, your databases.
- **On `dnf remove`**: payload + units are deleted; `/etc/duynhlab/`,
  `/var/log/duynhlab/`, `/var/lib/duynhlab/` and PostgreSQL data remain
  (see [`002-install.md` §6](002-install.md#6-remove) for full purge).
